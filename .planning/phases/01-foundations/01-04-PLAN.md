<objective>
Walk the project owner through registering the Azure AD app and submitting the Minecraft API permission form so Microsoft's 1-7 day review queue starts running during Phase 1 and 2 implementation. Phase 2 cannot complete without this approval — starting the queue in parallel with Phase 1 scaffold work is the explicit Phase 1 unblocker.

This plan is a pure external-action + documentation plan. No code changes. It covers Phase 1 success criterion 2 ("Azure AD app is registered with Minecraft API scope and has been submitted for Microsoft review; queue running; unblocks Phase 2") and addresses RESEARCH.md Pitfall 4 ("Azure app registered but MCE form not submitted"). Portal: https://entra.microsoft.com (or https://portal.azure.com → Microsoft Entra ID)

Step-by-step (RESEARCH.md §Azure AD App Registration Playbook):

1. Sign in to entra.microsoft.com with owner's PERSONAL Microsoft account (D-14 — same account owner uses for Minecraft)

2. Entra ID → App registrations → + New registration

3. Fill in:
   - Name: `WiiWho Client` (shown to users during device-code consent)
   - Supported account types: **Personal Microsoft accounts only** (D-15; this is the "consumers" tenant option)
   - Redirect URI: leave BLANK at this step

4. Click Register. Overview page shows Application (client) ID — copy this GUID.

5. Authentication (left sidebar):
   - + Add a platform → Mobile and desktop applications
   - Check `https://login.microsoftonline.com/common/oauth2/nativeclient` (canonical public-client native redirect)
   - Configure
   - Scroll to Advanced settings → Allow public client flows → set to Yes (REQUIRED for device-code flow per MSAL.NET docs)
   - Save

6. API permissions:
   - Confirm User.Read is present by default; leave it
   - DO NOT attempt to add XboxLive.signin via "Request API permissions" — that permission is NOT exposed in the standard Azure portal UI (verified via Microsoft Q&A). It is requested via OAuth scope at runtime.

7. Submit Minecraft API permission request form:
   - URL: https://aka.ms/mce-reviewappid (redirects to a Microsoft Forms URL)
   - Fields (owner fills interactively; Microsoft Forms loads dynamically, exact field list not machine-readable):
     - Application (client) ID (copied from Overview)
     - Tenant type: personal / consumers
     - App name and purpose: "Personal / small-group Minecraft 1.8.9 custom launcher"
     - Owner contact email: the email of the MS account that owns the Azure app
     - Expected user volume: low (personal + small-group)
   - Submit.
   - **Record submission timestamp immediately after clicking Submit** — this is the Microsoft review queue countdown start.

8. Store:
   - Client ID in `.planning/STATE.md` (NOT a secret per D-18 — public client, device code flow, no secret; but it's still easier to look up from STATE than hunting email threads)
   - Submission timestamp in `.planning/STATE.md`
   - A maintainer doc `docs/azure-app-registration.md` with full context (what was configured, what to do if approval is slow, how Phase 2 uses the client ID)
   - Approval confirmation email (arriving in 1-7 days at the owner's MS-account email) should be pinned / saved to a password manager for later reference

Known ambiguity (RESEARCH.md §Known ambiguity — device code flow tenant):
- Microsoft's MSAL.NET docs say AADSTS90133 on /consumers for device code
- minecraft.wiki says /consumers is REQUIRED for XboxLive.signin
- Empirical evidence (prismarine-auth, Lunar, Prism, Helios) says it works
- Resolution: D-15 locks consumers. Phase 2 re-verifies empirically. Phase 1 does not re-litigate.

Waiting period: 1-7 days typical; some cases run weeks. Phase 2 planning / discussion work can start before approval, but Phase 2's final integration (actually calling `api.minecraftservices.com/authentication/login_with_xbox`) will 403 until approval arrives.
