# Phase 2 — UI Design Contract

> **Scope:** LoginScreen, LoadingScreen (silent refresh), DeviceCodeModal, ErrorBanner, AccountBadge + dropdown. Out of scope: Play screen layout changes (frozen from Phase 1), Settings (Phase 3), crash viewer (Phase 3).

## Spacing Scale

Declared values (4-point base; all used values must be multiples of 4):

| Token | Value | Usage in Phase 2 |
|-------|-------|------------------|
| xs | 4px | Icon-to-label gaps inside buttons, badge ring offset |
| sm | 8px | DropdownMenu item vertical padding, inline banner icon gap |
| md | 16px | Default gap between stacked elements, modal content padding-x on mobile-sized content |
| lg | 24px | LoginScreen vertical rhythm (logo → button → version spacing base unit), modal content padding |
| xl | 32px | Modal outer padding (Dialog default), LoginScreen block margin |
| 2xl | 48px | LoginScreen logo-to-button gap (matches Phase 1 Play screen `mb-8` = 32px — use 32px not 48px to stay consistent with existing App.tsx) |
| 3xl | 64px | Not used in Phase 2 |

**Exceptions:**
- Device-code display uses monospace + `letter-spacing: 0.15em` — letter-spacing is typographic, not spacing-token.
- Account badge circular head is `32×32px` (size-8 Tailwind) — on the 4-point grid.
- Dropdown menu trigger hit target is `32×32px` minimum (passes 24px minimum touch target for pointer input; not a touch UI).
- Login "Log in with Microsoft" button uses Phase 1 Play button's proportions: `px-12 py-6` (48px × 24px) = on-grid.

## Typography

Locked to **4 sizes, 2 weights** (within the spec's hard cap):

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Body | 14px (`text-sm`) | 400 (regular) | 1.5 (`leading-6`) | Error banner message, dropdown menu items, modal helper text, UUID display |
| Label | 12px (`text-xs`) | 400 (regular) | 1.5 | `v0.1.0-dev` version string, UUID truncation hint, timer "expires in 14:32" |
| Heading | 20px (`text-xl`) | 600 (semibold) | 1.2 (`leading-tight`) | DeviceCodeModal title "Sign in with Microsoft", error banner heading if used |
| Display | 36px (`text-4xl`) | 700 (bold) | 1.2 | LoginScreen `Wiiwho Client` wordmark (matches Phase 1 `text-4xl font-bold`) |

**Device-code monospace exception** (one additional role, not counted in the 4):

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Code | 28px (`text-3xl`) | 600 (semibold) | 1.2 | The 8-character MS device user-code, rendered in `ui-monospace, SFMono-Regular, 'Cascadia Mono', Menlo, Consolas, monospace` — required for glanceable entry; monospace is functional not decorative. |

**Rationale for only 2 weights:** regular + semibold covers all copy. Bold (700) on the wordmark is inherited from Phase 1 `App.tsx` and is the *only* bold use in the app — treated as a one-off brand mark, not a typography role.

60/30/10 split against Phase 1's existing dark palette:

| Role | Value | Usage |
|------|-------|-------|

| Secondary (30%) | `#171717` (`neutral-900`) and `#262626` (`neutral-800`) | Modal surface (`neutral-900`), dropdown menu surface (`neutral-900`), inline error banner surface (`neutral-900` with `border-red-900/50`), button hover (`neutral-800`) |
| Accent (10%) | `#16e0ee` (`--color-wiiwho-accent`, Wiiwho cyan) | **Reserved for:** the Login button (D-05), the "Try again" primary action on error banners, the focus ring (via `focus-visible:ring-[--color-wiiwho-accent]`), the AccountBadge ring on hover, and (post-Phase-2) the Play button. **NOT** for: link hover underlines, dropdown item hover, help-link text, text emphasis, or any decorative use. |
| Destructive | `#dc2626` (`red-600`) — border only (`border-red-900/50` = `#7f1d1d` at 50%) and icon (`text-red-500`) | Error banner border + alert icon. **Intentionally no red surface** — red-600 as a full background is visually heavy and conflicts with "actionable, not scary" tone of D-08's inline banner. Error banner uses `bg-neutral-900` + `border border-red-900/50` + `text-red-400` for the title and `text-neutral-300` for body copy. |
| Text primary | `#e5e5e5` (`neutral-200`) | Default body/UI text |
| Text muted | `#737373` (`neutral-500`) | `v0.1.0-dev`, UUID prefix preview, "expires in" timer |
| Text on accent | `#0a0a0a` (`neutral-950`) | Text on `#16e0ee` cyan button — identical to Phase 1 Play button |

**Accent reserved for (explicit, non-negotiable list):**

1. LoginScreen "Log in with Microsoft" button (primary surface, `bg-[#16e0ee]`)
2. DeviceCodeModal "Copy code" button (primary action among the three modal buttons; the other two are `outline` / `ghost`)
3. ErrorBanner "Try again" button (primary action; "Help" is `ghost` / link variant)
4. Keyboard focus ring everywhere (`focus-visible:ring-[--color-wiiwho-accent]` — already on shadcn Button)
5. AccountBadge outer ring on hover / focus (thin 2px cyan ring)

**NOT accent:** regular text, link body text, dropdown item hover, modal heading, icon tints (except focus ring), success states (none in Phase 2).

**No success color in Phase 2** — successful login is communicated by transitioning screens (login → loading → Play-forward), not by a green toast. Matches D-03 and overall "quiet success, explicit failure" posture.

| Wordmark | `Wiiwho Client` (matches Phase 1 App.tsx exactly) |

No tooltips, no legal text, no "Why Microsoft?" explainer (explicitly deferred in CONTEXT.md Deferred Ideas).

### LoadingScreen (D-02, silent-refresh state)

| Element | Copy |
|---------|------|
| Heading | `Wiiwho Client` (wordmark only, same style as login) |
| Status | No text — a spinner (`Loader2` from lucide-react with `animate-spin`) beneath the wordmark |
| Version text | `v0.1.0-dev` (same position as LoginScreen — keeps layout stable across login→loading→play transitions) |

**Minimum visible duration:** 300ms. Prevents flash when the refresh completes in <100ms. If the refresh takes longer than 8s without resolving, fall through to LoginScreen (per D-03, no error surfaced on this path).

### DeviceCodeModal (D-06, D-07)

Structure: centered Dialog, width 480px (`sm:max-w-[480px]`), locked from Phase 1 `launcher/src/renderer/src/components/ui/` conventions.

**Active state (waiting for Microsoft):**

| Element | Copy |
|---------|------|
| Title | `Sign in with Microsoft` |
| Body paragraph | `Enter this code on Microsoft's sign-in page to finish logging in.` |
| Code label | `Your code` |
| Code block | `{device_code}` — 8-char monospace, `text-3xl`, `tracking-[0.15em]`, centered, selectable text |
| Copy button | `Copy code` (with `Copy` lucide icon, switches to `Check` icon + label `Copied` for 1.5s on click) |
| Open browser button | `Open in browser` (with `ExternalLink` lucide icon — calls `shell.openExternal(verification_uri)`) |
| Timer | `Expires in {mm:ss}` (counts down from Microsoft's `expires_in`, typically 15:00) |
| Cancel button | `Cancel` (outline variant, also bound to Escape per D-07) |

**Expired state (D-06):**

| Element | Copy |
|---------|------|
| Title | `Sign in with Microsoft` (unchanged) |
| Body paragraph | `The code expired before you finished signing in.` |
| Primary action | `Generate new code` (replaces the code block + both prior buttons; clicking re-runs the device-code flow in-place, returns the modal to the active state with a fresh code) |
| Secondary action | `Cancel` (outline variant, unchanged) |

**Keyboard contract:**
- `Escape` → Cancel (closes modal, stops polling, returns to LoginScreen) — native Dialog behavior
- `Enter` in active state → triggers `Copy code`
- `Enter` in expired state → triggers `Generate new code`
- Focus on open: the code block itself (for screen readers to announce the code), tab order: code → Copy → Open in browser → Cancel

### ErrorBanner (D-08, D-09, D-10)

Inline banner beneath the login button. Full width of the button column. Not a toast. Not a modal. Persistent until dismissed or next login attempt (D-08).

**Visual structure:**
```
[AlertCircle icon]  {message}                           [× dismiss]
                    [Try again]  [Help →]
```

**XSTS + entitlement error messages (verbatim from CONTEXT.md D-10, locked):**

| Error | Message | Help link |
|-------|---------|-----------|
| `XSTS 2148916233` (no Xbox profile) | `This Microsoft account doesn't have an Xbox profile yet. Create one at xbox.com and try again.` | `https://www.xbox.com/en-US/live` |
| `XSTS 2148916235` (country blocked) | `Xbox Live isn't available in your country, so your Microsoft account can't sign in to Minecraft.` | `https://www.xbox.com/en-US/legal/country-availability` |
| `XSTS 2148916236` / `2148916237` (age verification) | `Your Xbox account needs age verification before it can use Minecraft.` | `https://account.xbox.com/en-US/Profile?activetab=main:mainTab2` |
| `XSTS 2148916238` (child not in Family) | `This account is under 18 and needs to be added to a Microsoft Family group by an adult.` | `https://account.microsoft.com/family/` |
| No Minecraft Java Edition entitlement | `This Microsoft account doesn't own Minecraft Java Edition.` | `https://www.minecraft.net/en-us/store/minecraft-java-bedrock-edition-pc` |
| Unrecognized XSTS code | `Microsoft sign-in failed (code {N}). Try again, or click Help for more info.` | `https://support.xbox.com/` |
| Network error (D-12) | `Can't reach Microsoft — check your internet connection.` | (no Help link — suppress the Help button entirely on network-error banners) |
| User cancelled device-code flow (D-07) | *(no banner — silent return to LoginScreen)* | — |
| Generic / uncategorized main-process exception | `Something went wrong while signing in. Try again, or click Help for more info.` | `https://support.xbox.com/` |

**Banner element labels:**

| Element | Copy |
|---------|------|
| Primary action button | `Try again` (per D-09 — re-runs device-code flow from the start) |
| Secondary action | `Help` (opens Help link in external browser via `shell.openExternal`; hidden when no Help link, see Network error row) |
| Dismiss | `×` (ARIA `aria-label="Dismiss error"`) |

**Lifecycle:** Banner appears after a user-initiated login attempt fails. Persists until the user clicks `×`, clicks `Try again`, or clicks `Log in with Microsoft` again (auto-dismisses on retry). **Never** appears on the silent-refresh path (D-03).

**Badge (collapsed, default state):**

| Username | `{mc_username}` (right of avatar, `text-sm`, `neutral-200`, max 16 chars then ellipsis) |

**Dropdown menu (opens on click, per D-13):**

| Menu item | Copy |
|-----------|------|

### Skin avatar contract (D-14 Claude's discretion)

**Decision: use `mc-heads.net`** for the skin head PNG.

Reasoning: all three candidate services (`minotar.net`, `crafatar.com`, `mc-heads.net`) render the head correctly; `mc-heads.net` has:
- Clean URL pattern (`https://mc-heads.net/avatar/{uuid}/32`)
- HTTPS, CORS-permissive for `fetch()` from Electron renderer
- Serves `Cache-Control: public, max-age=86400` — browser cache aligns with our local-cache intent
- No auth, no query-string fingerprinting

Local cache (D-14 mandatory):
- Path: `%APPDATA%/Wiiwho/skin-cache/{uuid}.png` (Windows); `~/Library/Application Support/Wiiwho/skin-cache/{uuid}.png` (macOS)
- TTL: 7 days (skins change rarely; 7d matches the refresh-token window naturally)
- On cache hit: read file, load via `file://` URL in renderer `<img src>`.
- On cache miss or stale: fetch from `mc-heads.net`, write file, display. While loading: show `neutral-700` placeholder square with user initial in `text-xs neutral-200`.
- On fetch error (offline, 404, 5xx): show the initial-placeholder permanently for this session. Do NOT show an error banner.

Path + TTL are planner-owned implementation detail; the decision to use `mc-heads.net` and to cache-by-UUID is locked here.

### Destructive actions

Only one destructive action in Phase 2: **Log out**. Per D-15 there is **no confirmation dialog** — logging out is cheap to reverse by logging back in. This is intentional and load-bearing; do not add an "Are you sure?" step.

| Action | Confirmation approach |
|--------|----------------------|
| Log out | None — instant on menu-item click, per D-15 |
| Cancel device-code flow | None — closes modal, drops polling, no network-side cleanup needed (Microsoft expires the code naturally), per D-07 |

## Motion & Interaction

Minimal motion — matches the "quiet, consistent" tone.

| Surface | Motion |
|---------|--------|
| Login → Loading → Play transitions | No transition (fade OR instant — executor picks; if fade, `transition-opacity duration-200`). A transition slower than 200ms makes the app feel sluggish. |
| DeviceCodeModal open | shadcn Dialog default (fade + subtle scale, `data-[state=open]:animate-in` — already baked into the block) |
| DeviceCodeModal close | shadcn Dialog default |
| ErrorBanner appear | `transition-all duration-150` — slide from `opacity-0 translate-y-[-4px]` to `opacity-100 translate-y-0` |
| AccountBadge dropdown open/close | shadcn DropdownMenu default |
| Copy button "Copied" flash | 1500ms, icon swap `Copy → Check`, label swap `Copy code → Copied`, color unchanged |
| Spinner (LoadingScreen) | `Loader2 animate-spin` — Tailwind default 1s linear infinite |

**No parallax, no skeleton loaders, no shimmer effects** — this is a 650px-tall launcher window, not a content dashboard.

## Accessibility

Non-negotiable baseline (passes WCAG AA):

- **Focus ring:** `focus-visible:ring-2 focus-visible:ring-[--color-wiiwho-accent]` is already the shadcn Button default. Every interactive element (buttons, dropdown triggers, dismiss × , code block when copy-via-keyboard is implemented) must be keyboard-focusable and show this ring. **No `outline: none` without a replacement.**
- **Escape closes DeviceCodeModal** (D-07) — handled natively by shadcn Dialog's `onOpenChange`.
- **Contrast checks:**
  - `#e5e5e5` on `#111111` = 14.5:1 — AAA
  - `#16e0ee` cyan background with `#0a0a0a` neutral-950 text = 13.1:1 — AAA (both Play and Login buttons)
  - `#737373` neutral-500 on `#111111` = 4.6:1 — AA body (used for version text, UUID truncation — acceptable because it's always paired with a higher-contrast primary near it)
  - `text-red-400` (`#f87171`) on `#171717` (neutral-900 banner surface) = 5.2:1 — AA
- **Screen reader:**
  - Device-code block uses `aria-live="polite"` and `aria-label="Your sign-in code: {code_with_spaces_every_4}"`, ensuring the 8-char code is read as individual characters, not as a mis-tokenized word.
  - Error banner uses `role="alert"` so its appearance is announced.
  - AccountBadge dropdown trigger has `aria-label="Account menu for {username}"`.
- **Reduced motion:** Respect `prefers-reduced-motion`. For Phase 2 this reduces to: spinner stops rotating (substitute `...` text or hold a static glyph). ErrorBanner and Dialog skip their enter/exit animations. Tailwind's `motion-safe:` prefix is the mechanism.
